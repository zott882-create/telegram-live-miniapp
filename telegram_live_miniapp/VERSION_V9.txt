V9.81 — stability patch on top of V9.80

1. render.yaml: plan free → starter (free засыпает после 15 минут,
   что обнуляет всю архитектуру с очередью Telegram-сообщений).

2. Убрана дублирующая строка goal_wait["enabled"] = False в
   _handle_permanent_telegram_failure (стр. 5879-5881). Раньше она
   выполнялась после if'а и обнуляла changed-флаг — после блокировки
   бота пользователем запись на диск могла не произойти.

3. Per-row save вместо записи всей таблицы при каждом изменении:
   - добавлены _save_one_notify_sub() и _save_one_notify_matches()
   - добавлен _delete_notify_row() для точечного DELETE
   - 11 горячих путей переведены: handle_subscribe, handle_notify,
     handle_goal_subscribe, handle_goal_wait_subscribe,
     handle_notify_matches_clear, handle_notify_match_remove,
     _admin_delete_user, _admin_block_user,
     _handle_permanent_telegram_failure, _scan_notify_subscription_once,
     telegram_send_worker_loop
   - на 1000 подписчиков успешное Telegram-сообщение теперь делает
     1 UPSERT вместо 1000.

4. requirements.txt: убран неиспользуемый requests (в коде только urllib).

5. main(): обработчик SIGTERM + graceful drain очереди Telegram.
   Render шлёт SIGTERM перед kill за ~30 секунд. Раньше main()
   ловил только KeyboardInterrupt — SIGTERM убивал процесс на
   середине, daemon-потоки обрывались, сообщения из очереди
   терялись. Теперь даём 5 секунд на дослание + persistent
   jobs подхватятся на следующем старте.
