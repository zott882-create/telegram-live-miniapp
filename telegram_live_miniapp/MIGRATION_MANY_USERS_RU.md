# Миграция под много пользователей

В этой версии состояние пользователей переведено с JSON-файлов на SQLite.

## Что мигрировано

Раньше уведомления хранились в файлах:

- `data/notify_subs.json`
- `data/notify_matches.json`

Теперь используется база:

- `data/notify_state.sqlite3`

При первом запуске приложение автоматически переносит старые JSON-данные в SQLite. Пользовательские фильтры, профили, найденные матчи, seen/cooldown и подписки на голы сохраняются.

## Почему это важно

JSON-файл плохо подходит для большого количества пользователей: каждый апдейт перезаписывает файл целиком, при параллельных запросах выше риск потери данных. SQLite работает атомарно, использует WAL и лучше держит одновременные чтения/записи на одном сервере.

## Настройки Render

Для обычного запуска ничего менять не нужно. По умолчанию:

```env
NOTIFY_STORAGE=sqlite
NOTIFY_MIGRATE_JSON=1
```

Если нужно временно откатиться на старые JSON-файлы:

```env
NOTIFY_STORAGE=json
```

## Важно для Render

На бесплатном Render локальные файлы могут быть непостоянными после пересборки. Для реального продакшена лучше подключить Persistent Disk или перейти на PostgreSQL.

Минимально правильный вариант на одном инстансе:

- `NOTIFY_STORAGE=sqlite`
- Render Persistent Disk, смонтированный в папку `data/`
- один web-инстанс

Для большого количества пользователей и нескольких инстансов лучше делать следующий шаг: PostgreSQL для `notify_subs`, `notify_matches`, избранного и пользовательских настроек.

## Что осталось общим кэшем

Live-матчи уже читаются из общего collector-кэша:

- `data/live_cache.sqlite3`

То есть 100 пользователей не должны создавать 100 одинаковых запросов к IGScore. Один collector обновляет данные, пользователи читают кэш.

## V9.42: PostgreSQL

Для 500 пользователей лучше использовать PostgreSQL вместо SQLite для пользовательского состояния. В этой версии `NOTIFY_STORAGE=auto`: если есть `DATABASE_URL`, будет использоваться Postgres; иначе останется SQLite.

Подробно см. `POSTGRES_RENDER_RU.md`.
